## 1. Why this exists

The **proprietary in-trip behavioral data graph** is the product's primary long-term moat: a growing, structured record of *"what's actually good right now"* — which side quests travelers did, loved, or skipped; when; in what weather, time, and mood; for what kind of traveler. It compounds with every trip, a generic LLM cannot replicate it, and the assistant-mode product generates it naturally. This brief designs and builds the **schema and persistence architecture** for that graph — the stable data contract everything downstream depends on.

**Three consumers the schema must serve (design for all three now, build only the schema):**
- **Personalization/memory** — per-user taste profile driving recommendations and retention lock-in.
- **Fine-tuning (#1)** — a `TrainingExample` must derive from captured events with minimal transform (see the sibling brief). The behavioral event is the canonical upstream record; training data is one projection.
- **Ground-truth network (#8)** — crowd inputs need **trust/confidence scoring**, **staleness decay**, and **poisoning resistance**. The schema must carry the fields those mechanisms need (source, timestamp, trust signal), even though the algorithms are built later.

**Sensitivity (ties to #7):** this graph holds solo-traveler **location and behavior** data. PII/consent handling is a first-class schema concern, not an afterthought — carry consent state, a de-identification seam, and a privacy classification on every record from day one.

## 2. Objective & definition of done

- A **conceptual model** (entities, edges/relationships, event taxonomy) approved by the human, captured as **ADR-0010** (data-model decision, incl. graph-vs-relational rationale).
- **Domain layer:** the behavioral-graph **entities and value objects** as `attrs` classes in `app/domain/models/` — pure, no framework/I-O imports. A canonical **behavioral event** type rich enough to (a) project to `TrainingExample` and (b) support trust/staleness fields for #8.
- **Ports:** repository/persistence **Protocols** in `app/domain/ports/` (Protocol-first per `ports.md`), with attrs request/response types.
- **Adapter:** **SQLAlchemy 2.0** ORM models + repository implementation in `app/adapter/output/`, mapping ORM↔domain (domain never imports SQLAlchemy), plus a **mock adapter** per service (repo convention).
- **Migrations:** **initialize Alembic** (`uv run alembic init alembic`, wired to settings/DB session per ADR-0009) and generate the **first autogenerated migration** for the schema.
- **DI wiring:** repository provider added to the `dependency-injector` container (ADR-0009), injected via `Provide[Container.x]`.
- **Tests:** domain invariants + mapping round-trip (domain→ORM→domain) + mock-adapter tests, pytest naming per `tests.md`; mark any DB-touching test `@pytest.mark.integration`.
- **Done =** ADR-0010 merged; domain models + ports + SQLAlchemy adapter + mock + Alembic migration committed; all five checks green (`pytest`, `ruff` x2, `lint-imports`, `basedpyright`); a `TrainingExample`-projection function (or documented mapping) proves the alignment; `DECISIONS.md` records every choice.

## 3. Scope guardrails (non-goals)

- **No recommendation/personalization *algorithms*, no trust-scoring or staleness-decay *logic*, no ingestion pipeline.** Design the schema to *support* them; do not build them. Stubs/interfaces only where a seam is needed.
- **No agent/LLM changes, no new API endpoints** beyond what's needed to prove wiring (a thin write path is optional and `[HUMAN DECISION]`). Do not modify the on-device dirs (`explore/`, `bench/`) or `frontend/`.
- **No graph database introduced without explicit approval** — default is relational modeling of graph structure in the existing SQLAlchemy/Postgres stack (bootstrapper simplicity). Escalate if you believe a graph DB is warranted (`[HUMAN DECISION]`).
- **Do not break existing chat endpoint/tests.** Keep the hexagonal boundaries clean (import-linter must stay green).

## 4. Architecture & placement (hexagonal — strict)

Follow ADR-0005. Dependencies point inward only.

- `app/domain/models/` — behavioral-graph entities & value objects (`attrs`, `@define`; `pydantic` only if a boundary needs it). **No SQLAlchemy, no I/O.** This is the canonical schema.
- `app/domain/ports/` — repository `Protocol`(s) (e.g. `BehavioralEventRepository`, `TasteProfileRepository`), attrs request/response objects, Protocol-first ordering per `ports.md`.
- `app/adapter/output/` — SQLAlchemy ORM models + repository implementation (maps ORM↔domain), plus a **mock repository** for tests. ORM models are an adapter detail; the domain never imports them.
- `app/infrastructure/` — Alembic setup, DB session/engine, settings, and the DI container provider wiring the concrete repo to the port. The router→container `Provide[...]` seam is the only sanctioned outward import.
- **Migrations:** Alembic lives at repo/backend per `CLAUDE.md`; autogenerate from the SQLAlchemy models, review the generated SQL before committing.

## 7. Key design questions to resolve with the human (`[HUMAN DECISION]`)

1. **Graph vs relational** — relational modeling in Postgres (default, bootstrapper-simple) vs a dedicated graph store. Recommendation: relational now; revisit only if traversal patterns demand it.
2. **Entity & edge set** — the node types and the interaction verbs (did/loved/skipped/saved/...), and what counts as context (time, geo, weather, mood, traveler-type).
3. **Event granularity & event-sourcing** — append-only event log as source of truth with derived read models, vs mutable current-state tables. (Append-only aligns best with training-data derivation and staleness.)
4. **PII/consent model** — per-record privacy classification, consent state, de-identification seam; how location precision is stored/coarsened.
5. **Trust & staleness fields** — what source/confidence/decay inputs to carry now so #8 can score later without a schema migration.
